Source of Truth And Global Rules

The Mentoria Hub product rules in AGENTS.md and the current app structure remain the source of truth. The app must continue to feel like a working EdTech product for students in grades 8-11 from Kazakhstan and global contexts, while also staying credible to schools, partners, and sponsors.

  • Use the Mentoria cybercore and digital ethnofuturism direction: precise route nodes, woven-grid motifs, dark technical surfaces, and restrained luminous accents.
  • Use Mentoria palette tokens: #07131E, #030B12, #11C6CF, #58D9EF, #84E6EA, #1594B9, #1676B8, and #315A75.
  • Do not use purple gradients, generic neon glow, beige-only palettes, stock-like atmospheric imagery, decorative blobs, or emoji.
  • Use premium sans-serif stacks such as Geist, Satoshi, Outfit, Avenir Next, or system UI. Do not introduce Inter as a named project font.
  • Use min-height: 100dvh for full-screen sections.
  • Motion must use transform and opacity only, with tactile CTA and chip states.
  • Respect prefers-reduced-motion.
  • Do not nest UI cards inside other cards. Product consoles may contain rows, chips, fields, and status modules, but avoid decorative piles of cards.
  • All fixed-format UI elements need stable dimensions or responsive constraints, so labels, hover states, and dynamic content do not shift layout.

Design References

The following named skills are design guidance for future implementation work. They should inform interaction quality, visual refinement, and material decisions, but they do not override AGENTS.md, the existing app architecture, or the exact product requirements in this context.

design-taste-frontend

Use its guidance for high-agency frontend decisions: dependency verification, no emoji, stable responsive layouts, premium sans typography, full loading, empty, and error states, tactile feedback, and GPU-safe transform/opacity motion.

high-end-visual-design

Use it for high-end interaction polish, clear spatial rhythm, refined surfaces, careful hover and active states, and avoiding generic layout or typography defaults. Adapt its ideas to Mentoria's dark cybercore palette instead of changing the product direction.

ecc:liquid-glass-design

Use the Liquid Glass idea only as a material reference for selective glass surfaces: subtle inner borders, readable contrast, restrained blur, and interactive glass only where the element is truly interactive. Avoid applying glass to every surface.

1. Hero Page Changes

1.1 Course preview block

The hero page must keep the section title Free courses that support the journey. The visible hero course preview must be reduced to exactly three courses.

  • Remove the Chemistry course from the hero course preview.
  • Remove the University Admission course from the hero course preview.
  • Show exactly three course preview items.
  • Make those three preview slots fully scalable so course covers can be inserted later without redesigning the component.
  • Course previews in this hero block must not be interactive. They are supporting signals, not clickable course cards.
  • Remove the console title Courses library.
  • Remove the supporting console copy Focused course tracks support the opportunity journey..
  • Remove the search placeholder/copy Search Mentoria courses.
  • Replace the removed console copy with the exact copy Try out one of our best courses!.
  • Remove track labels above each course name, including labels such as Mentoria English, Mentoria Physics, and any similar Mentoria ... label.

Hero preservation rules

  • Do not change the actual hero alignment unless explicitly requested later.
  • The first viewport must continue to feel like a working EdTech product, not a generic landing page.
  • The primary hero function remains opportunity search and matching.
  • Courses may appear only as supporting signals around the opportunity-matching workflow.

2. Workspace And Auth Loading Changes

Loading screen copy

  • For the workspace loading process, remove the text Mentoria Hub above Loading student workspace.
  • The loading screen should show Loading student workspace without the extra brand label above it.
  • Create a separate loading screen with the exact text Loading onboarding questions.
  • Loading onboarding questions appears only after successful new registration.
  • The Loading onboarding questions screen must visually match Loading student workspace.

Existing-account infinite loading bug

There is a current failure when logging into an already existing account: the app can stay in infinite workspace loading with the Supabase error new row violates row-level security policy for table "profiles". This must be documented and fixed in implementation.

  • The app must not attempt to insert a new profile row for an existing account during normal signin.
  • Existing-account signin should fetch the existing profile and continue to the dashboard.
  • Profile insert/upsert logic must be compatible with the active Row Level Security policies.
  • When an RLS failure occurs, the UI must not stay in an endless loading state.

Registration and onboarding order

This registration and onboarding order change is already implemented in the app and is excluded from the pending corrections scope for this context.

  • Start your journey already opens registration/login instead of launching onboarding automatically.
  • New signup with new data already opens onboarding after the loading transition.
  • Signup with existing data already shows an existing-account notice.
  • Signin with valid data already opens the dashboard after loading.
  • An existing browser session already opens the dashboard immediately after Start your journey or Log in.
  • Do not duplicate this flow as pending work in future correction passes unless a regression appears.

3. Dashboard Changes

Course and opportunity state

  • When a newly created account enters the dashboard, it must not be assigned all existing courses automatically.
  • New accounts must start with zero enrolled courses and zero active courses.
  • Stop any reading of user-owned dashboard or course state from the HeroPage.
  • HeroPage data is static marketing/product-preview data only. It must not populate student-owned dashboard state.
  • Dashboard state must come from user-owned storage such as profile-linked enrolled courses and saved/chosen opportunities.

Metrics strip

Replace the existing large rectangular metric cards for Active courses, In progress, Recommended matches, and Upcoming deadlines.

  • Use one compact single-row data strip that takes much less space.
  • Keep only Active courses and Chosen opportunities.
  • Under Active courses, keep a View button.
  • The Active courses View button navigates to My courses.
  • Under Chosen opportunities, keep a View button.
  • The Chosen opportunities View button navigates to My opportunities, which is the renamed opportunity search workspace.

Learning section

  • In the dashboard Learning section, do not display course rows if the user has no enrolled courses.
  • If the user has no enrolled courses, show the exact empty state copy It seems like you don’t have anything to learn.

Recommended opportunities temporary window

  • Rename Shortlist to Recommended opportunities.
  • Make the recommended opportunities area a temporary grayish window or tab.
  • The window must have a close button in the top-right, visually like a small dismissible window.
  • Include the subtle bottom description we’ve found something you might like from your answers to onboarding questions.
  • The window always contains exactly three opportunities.
  • The three opportunities must be selected by recsys based on onboarding question answers.
  • When the user adds or saves opportunities from this window, the window closes automatically.
  • After the user manually closes this window, it must never appear again for that user.
  • At the bottom of the window, offer a path to opportunity search and a path to ask MentorLM for help finding opportunities.

4. Legacy MentorPet Rename Requirement

All legacy MentorPet, Mentor pet, mentor-pet, and MentorPet-related concepts must be renamed to MentorLM. The old name may appear in implementation notes only as a migration source term.

  • Navigation labels must use MentorLM.
  • Lesson notes must be called MentorLM notes.
  • Companion, helper, popup, workspace, note, route, and data names should migrate to MentorLM naming where practical.
  • Any user-facing text still using the old name must be translated to English and renamed.

5. My Courses Page And Course Catalog

Empty state and search

  • If the user has no enrolled courses in My courses, show the exact copy Oops, there are no courses. Would you like to find some?.
  • Under that empty state, show a searchbar that is almost full-width, matching the width of the other long workspace elements.
  • The searchbar belongs to the course catalog/feed on the same page.

Course catalog/feed

On the same page, under the searchbar, users can find courses, including the current five existing courses. This area is effectively the course catalog previously described by the product direction.

  • The course catalog/feed must appear even when the user's enrolled course list is empty.
  • The feed must include the current five courses.
  • Courses should be shown in random order under the searchbar.
  • Each course should appear as one full-width row, with the same width as the searchbar and other long elements.
  • Each row shows a short course description.
  • Each row shows a course cover on the left as a rectangular image area.
  • The cover must not span the full width of the course row.
  • On hover, the course row expands in the same spirit as the current English course preview expansion.
  • Remove the Hide option.
  • Remove labels above the course cover or title such as Mentoria English, Mentoria Physics, and similar track labels.

Course page

  • The course page must be fully scalable and editable from an admin panel in the future.
  • The course catalog/search feed must allow navigation to the course page.
  • At the top of the course page, add an unbordered back arrow that returns to course search/catalog.
  • Under the course cover, add the button enroll this course.
  • For every lesson row, replace the triangle/play button with a right-side button labeled go to the lesson.

6. Lesson Page Requirements

Scalable admin-editable lesson structure

  • The lesson page must be fully scalable and editable from an admin panel in the future.
  • The future admin panel must support adding a video to each lesson.
  • The video source can be a Telegram lesson link or a YouTube lesson link.
  • Remove the lesson cover from the lesson page.
  • Keep a video placeholder in the cover's place.
  • The video placeholder should become the video player area when a video URL exists.

Strict order below the video

The following sections must appear strictly under the video area.

1. Assignment

  • The assignment block must become an active call-to-action area, not a passive text block.
  • Add a textarea so the student can write an answer directly in the lesson page.
  • Also support an Attach file action where file upload is needed.
  • Add a bright Submit assignment button.
  • Assignment prompts, accepted file behavior, and submission settings must be available to mentors/admins in a dedicated admin panel tab later.

2. Materials

  • Do not show materials as comma-separated text.
  • The current comma-separated style looks like a draft and does not communicate whether items are files, links, downloads, or topic names.
  • Make materials clickable and visually clear.
  • Convert materials into clean cards or list rows with icons.
  • For example, show a document icon near Vocabulary bank and a download icon near Practice worksheet.
  • On hover, materials must clearly feel clickable.
  • On click, downloadable materials should start downloading.
  • Materials must be editable from the admin panel and scalable for any number of materials.

3. Check

  • The current check block asks a question but gives no way to answer it.
  • Turn the block into an interactive self-check mini quiz.
  • The quiz has three questions shown sequentially.
  • After the student answers one question, the next question appears.
  • Question answers can use radio buttons or a text input, depending on the question type.
  • Add a Check button.
  • After pressing Check, the interface immediately highlights the answer green for correct or red for incorrect.
  • At the end, show centered statistics such as 1/3, 2/3, or 3/3 correct.
  • At the end, also show a small supportive comment for each possible score value.

MentorLM notes on lesson pages

  • If no MentorLM note has been generated for the current lesson, remove the dedicated notes area.
  • If no note exists, leave only a mini MentorLM window.
  • If a note exists, keep the design unchanged and fill the MentorLM response into the note window.
  • MentorLM notes must be hideable.

7. Opportunity Search Renamed To My Opportunity

Workspace naming and layout

  • Rename the opportunity workspace concept to My opportunity.
  • Inside that workspace, show a centered window titled Opportunity search.
  • Copy the hero-style search UI into this workspace without changing the actual hero.
  • The copied search UI must be interactive.
  • The search window should be centered and visually aligned with the established Mentoria workspace design.

Search and match behavior

  • When the search field is empty, show the same opportunities that currently appear when opening this section.
  • When the search field is empty, do not show a match icon on any opportunity.
  • When the user types manual text into the search field, do not show match icons on results.
  • Show match icons only when the user uses filters and the system returns matching opportunities through the filter logic.
  • Opportunity results must be scrollable.

Saving and dashboard connection

  • At the bottom of every opportunity card, add a save button.
  • Saving an opportunity updates the dashboard.
  • Saved opportunities count as saved/chosen opportunities.
  • Saved/chosen opportunity count feeds the Chosen opportunities dashboard metric.
  • Under the opportunity search window, show a saved opportunities window.

9. MentorLM

Rename and integration

  • Connect the existing legacy companion functionality into MentorLM.
  • Rename every legacy companion-related item to MentorLM.
  • Redesign this section according to the established design rules while preserving the main structure and general code shape.

10. Public Interfaces And Data Notes

Separate static preview data from student-owned state

  • Student-owned state must be separated from static hero and course preview data.
  • Hero course previews must not create enrolled courses.
  • Hero opportunity previews must not create chosen opportunities.
  • Dashboard state must be derived from user-owned data, not from the homepage preview arrays.

Persistent user-owned storage

The following user-owned state needs persistent storage linked to the signed-in student profile:

  • Enrolled courses.
  • Active courses.
  • Saved or chosen opportunities.
  • Dismissed state for the Recommended opportunities temporary window.
  • MentorLM lesson notes.

Onboarding and recommendations

  • Onboarding answers are profile information.
  • Onboarding answers are used only for recommendations.
  • Recommended opportunities must be selected by recsys using onboarding question answers.
  • The temporary Recommended opportunities window must contain exactly three recsys-selected opportunities.

Admin-scalable structures

Course, lesson, assignment, material, video, quiz, and note structures must remain scalable for future admin editing.

  • Courses need editable titles, descriptions, covers, lessons, and enrollment settings.
  • Lessons need editable title, description, video source, assignment, materials, quiz, and MentorLM note configuration.
  • Materials need a structured shape, not plain strings joined by commas.
  • Quizzes need multiple questions, answer choices or text answers, correct answer metadata, and feedback text.
  • Videos need source type support for Telegram and YouTube links.
  • Mentor/admin tooling needs a dedicated area for assignment management.
