Executive Context

Mentoria Hub moves Mentoria from synchronous, Telegram-heavy operations and live video sessions toward a scalable asynchronous EdTech platform. The MVP must unite two core domains: a searchable catalog of educational opportunities and an asynchronous learning system for consultations and courses.

The platform is primarily for students in grades 8-11 globally, with a specific focus on Kazakhstan. It should help students discover hackathons, scholarships, internships, summer schools, academic programs, and structured Mentoria courses while giving Mentoria an operational dashboard for growth.

The architecture must separate frontend presentation from backend data, authentication, recommendation, and embedding services. The MVP should demonstrate real-world usefulness and international scaling potential.

Relational Data Modeling

The database must use strict relational modeling for all fields that are frequently queried, filtered, joined, sorted, or aggregated. Avoid placing core domain fields inside broad JSONB blobs because large JSONB documents can be TOASTed, reducing planner visibility and slowing scans on nested keys.

JSONB is allowed for dynamic translations and unpredictable auxiliary metadata, but not as the primary structure for events, users, enrollments, lessons, or progress.

Table Core Columns Purpose
profiles id uuid, role text, interests text[], grade_level int, preferred_locale text Public user profile linked one-to-one to auth.users; stores academic signals for recommendations.
events id uuid, title text, category text, deadline date, embedding vector(1536) Catalog of hackathons, scholarships, internships, summer schools, and opportunities.
saved_events user_id uuid, event_id uuid, saved_at timestamptz Many-to-many bookmark table with a composite primary key.
consultations id uuid, title text, description jsonb, difficulty text Master records for asynchronous courses and mentorship modules.
lessons id uuid, consultation_id uuid, video_url text, position int Instructional content linked to a course or consultation.
enrollments user_id uuid, consultation_id uuid, progress_pct float Tracks student enrollment and aggregate progress.
lesson_completions user_id uuid, lesson_id uuid, completed_at timestamptz Granular immutable completion records used to update enrollment progress.

Use strict foreign keys and cascading deletes where appropriate to prevent orphaned records when legacy events, courses, or lessons are removed.

Identity, RBAC, and RLS

Supabase Auth is the identity provider. PostgreSQL Row Level Security is the authorization boundary and must be enabled on every table in the public schema. Application checks can improve UX, but database policies are the source of truth.

Create a PostgreSQL trigger on auth.users that inserts a matching public.profiles record after signup. Extract safe initial metadata such as first name, last name, and preferred language from the auth metadata payload.

Target Action Policy Logic
profiles SELECT Users can view their own profile; admins can view all profiles. Example: auth.uid() = id OR is_admin().
profiles UPDATE Users can update only their own preferences, grade level, and locale.
events SELECT The event catalog is publicly readable unless later product requirements make it authenticated-only.
events INSERT / UPDATE / DELETE Only admins can modify the event and opportunity catalog. Example: is_admin().
saved_events ALL Users can access only rows where auth.uid() = user_id.

Use a locked-down user_roles table for role administration, but avoid joining it in every RLS policy. Instead, implement a Supabase Custom Access Token Auth Hook that injects user_role into JWT claims. RLS helpers can then check auth.jwt() ->> 'user_role' for admin, mentor, or student access.

Security constraint: do not use broad shortcuts such as email-domain based admin access or blanket RLS bypasses for normal application flows.

Asynchronous LMS Flow

The LMS should decouple educational progress from live instruction. Courses or consultations contain ordered lessons with multimedia content, materials, and progress checkpoints.

  1. A student starts a course through a Server Action that creates an enrollments row.
  2. The student completes lessons through Server Actions that insert timestamped lesson_completions records.
  3. A database trigger or materialized SQL view recalculates enrollments.progress_pct whenever completion state changes.
  4. The dashboard reads precomputed enrollment progress instead of aggregating lesson completion records on the client.

For MVP video delivery, store external URLs in lessons.video_url and render them through iframe-based embed components. Keep the rendering abstraction ready for future Supabase Storage or CDN-backed video delivery.

Quizzes, assessments, and surveys should be validated and graded server-side through Server Actions to prevent client-side manipulation before progress, certificates, or completion state are updated.

AI Recommendations and Hybrid Search

Recommendations should solve the cold-start problem by using explicit onboarding signals and semantic event data rather than collaborative filtering. Students provide grade level, interests, subjects, and goals. Events and courses carry structured metadata plus generated embeddings.

Search Method Technology Implementation Focus
Semantic Search pgvector and OpenAI embeddings Use vector(1536) columns and cosine distance with the <=> operator. Add HNSW indexes for performance.
Full-Text Search PostgreSQL tsvector and tsquery Use generated tsvector columns and GIN indexes for exact keywords, acronyms, and named opportunities.
Result Fusion Reciprocal Rank Fusion Implement a PostgreSQL search function with CTEs that merges semantic and keyword ranks into one ordered result set.
Embedding Pipeline Supabase Edge Functions, pg_net, and pg_cron Generate embeddings automatically after insert or update, and retry scheduled failures without an external worker system.
score = 1 / (60 + rank_semantic) + 1 / (60 + rank_keyword)

The query embedding can be generated by the Next.js server and passed to PostgreSQL, or generated through a controlled backend function. Avoid exposing OpenAI keys to the browser.

Localization Architecture

The i18n strategy is split between static interface localization and dynamic database content localization.

Content Type Storage Rendering Rule
Static UI text Codebase JSON dictionaries such as en.json, ru.json, and kk.json. Use next-intl middleware and locale-based routing to render translated UI in Server Components.
Dynamic event and course text JSONB translation columns such as title_translations and description. Read the active locale and select the matching JSON key, with a deterministic fallback locale.

Do not store admin-editable event or course translations in repository dictionaries because that would require code changes and redeployments for content updates. Do not create language-specific columns such as title_en, title_ru, and title_kk unless a later migration explicitly chooses that tradeoff.

Admin Dashboard and Performance

The admin dashboard should use the same Next.js application and the same business logic layer as the student-facing product. Admin Server Actions must validate JWT role claims, validate input with Zod, and then perform CRUD operations for events, courses, lessons, and user roles.

Performance should favor server-rendered HTML and minimal browser JavaScript. Use React Server Components for data-heavy views, Vercel for deployment and edge delivery, and CI checks for type safety and database migration validity before deployment.

  • Admin CRUD: events, consultations, lessons, and role management.
  • Student views: opportunity catalog, recommendations, saved events, enrolled courses, progress, and deadlines.
  • Operational target: fast loading on lower-end mobile devices and unstable network conditions.

MVP Architecture Defaults

  • Deploy frontend on Vercel.
  • Use Supabase CLI for local database development and migrations.
  • Use Supabase Auth with profile trigger synchronization.
  • Use RLS and JWT role claims for authorization.
  • Use normalized PostgreSQL tables for core entities.
  • Use JSONB only for dynamic translations and flexible metadata.
  • Use pgvector, tsvector, HNSW, GIN, and RRF for search and recommendation relevance.
